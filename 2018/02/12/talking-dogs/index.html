To make CheerUpCharm a little more interactive and a little more fun for the user...

2018-02-12
#code, interactivity, chat

To make CheerUpCharm a little more interactive
and a little more fun for the user, we considered adding a chat function. However, owing to the sensitivity of information that might be transmitted (since the app itself is meant to help people who may have deeply personal problems), actual chat functionality wherein the user could interact with another human user was largely unfeasible, especially if said users were anonymous. There would be too many problems that would be too difficult or time-consuming to address, particularly if harassment or abuse arose from it. If a human-to-human chat function were added, the only implementation of it that would work safely for the user would be if the user were only allowed to connect to a professional health care worker.

However, we did think it might be fun for the user to instead interact with a chatbot, something that would simply listen and respond in kind, perhaps offer some trivia or answer simple questions.

BotUI
was what we ended up experimenting with in our initial attempt at building an interactive chatbot. BotUI already has a chat-like interface designed for it so that was something we needn’t concentrate on for the moment, allowing us to focus on the actual bot-building.

BotUI bills itself as “super easy to create conversational/bot interfaces”, and to its credit, it really is quite simple to build a chatbot using it. At its very base, it applies two concepts to its chatbot-building, “Messages” and “Actions”.

Messages are, simply put, hardcoded text messages that the chatbot issues to the user following a predetermined flow (selected based on user responses). For example, in the small test that we created,

simply displays the question “How are you?” to the user.

Actions are activities that are performed depending on the user response; BotUI uses then() to allow chaining following responses and actions to continue the conversation:

The callback function defined inside then() is only displayed once the preceding message is displayed, and in this case that would be two buttons (using action.button) - one displaying the text “I am Good!” and the other “I am tired…”. Based on the button selected by the user, the chat responds with another message and continues chaining then()s.

Simply put, each part of the conversation using BotUI can be represented in “blocks”:

BotUI offers just two types of Actions: either a text entry response from the user, or button selection. Notably, it only offers one of these two options at any given time - they cannot be used at the same time in the same single response.

We found BotUI extremely simple to use, but that also came with its own caveats. Because of its response chaining structure, every response the user made would have to be predetermined, which offers the user very little interaction beyond the first few times they initiate an interaction with the bot. This also means creating a complex conversation chain would be far too time-consuming and unwieldy.

Futhermore, because the bot itself does not recognise user inputs as anything meaningful in and of themselves, it is unable to respond in a manner that would imply intelligent interpretation, even from a grammatical standpoint - in the sample, this error can be seen quite easily.

The user’s text input was “reading”, but the chatbot’s response was very clearly a hardcoded “You did text-input“. The bot would therefore be unable to remember or apply contextual information further down the conversation chain, as each part of the conversation is designed in message-action(response) blocks.

And – perhaps the most problematic part of creating this chatbot with BotUI – is simply that as the chat chain grows and becomes more complex, it would become entirely too difficult (or perhaps just annoying) to follow the conversation in terms of structuring the code as it would become quite lengthy, even for a short conversation.

However, despite BotUI being perhaps a bit on the simplistic side, it gave us an idea of how a basic chatbot functions. In order to create a more responsive/intelligent bot, we have plans to look into Google’s Cloud Machine Learning APIs (Dialogflow).

Like BotUI, Dialogflow is meant to create an interactive experience for a user, but unlike BotUI, which expects hardcoded Messages and Actions, Dialogflow is far more flexible - it operates on two concepts called “intents“ and “contexts“ to build responses. It is perhaps building a chatbot from the exact opposite perception of what the experience should be: rather than giving the user a preselected list of actions to use as responses, API.AI attempts to predict what or how the user will respond.

Of course, it still needs a list of predetermined responses, but rather than forcing the user to indicate a specific response (such as with BotUI’s buttons), Dialogflow uses machine learning to guess at and learn alternate responses from the user that carry the same meaning as what is already prerecorded, and use this information to respond appropriately.

“Contexts” also allow the chatbot to retain previously received information from the user and use this information later in the conversation, which preserves the integrity of the conversation - the user can continue to interact with the chatbot on a particular topic rather. In everyday human conversation, people frequently drop several words or replace them with vague terms (for example: “How was your day?” “It was good,” wherein “it” is a vague represention of the idea “your day”; it is rather unnatural for native speakers to repeat the topic as people remember it and interpret responses in that context).

For our project, it might be more reasonable for us to stick with a simple chatbot in order to avoid getting caught up with something rather too complex for our purposes. Either way, it’s all very exciting stuff.
